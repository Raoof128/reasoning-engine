# Reasoning Engine MCP Audit - 2026-06-21

Scope: single-agent production, security, reliability, and maintainability audit of
`/Users/raoof.r12/Desktop/Raouf/reasoning-engine`.

Note: the formal Codex exhaustive security-scan workflow requires explicit approval
to use subagents. This report is a manual single-agent audit of all tracked source,
tests, docs, and CI files in this compact repository.

## Executive Summary

The project is a small FastMCP Python server with a clear architecture, parameterized
SQLite access, and useful unit coverage for the deterministic scoring modules. The
main production risks are not SQL injection or arbitrary code execution; they are
trust-boundary problems around MCP tool inputs, unbounded data persistence, weak
runtime invariants, dependency pinning, and an orchestrator-enforced rather than
server-enforced prompt-injection boundary.

The highest-value next work is to add a validation layer for every MCP tool input,
persist budget changes atomically, make dependency versions reproducible, harden
stored memory/trace handling, and expand tests to cover malicious and malformed
inputs.

## Remediation Update

The main findings in this report were remediated after the audit:

- MCP tool inputs now pass through shared validation helpers.
- Score, confidence, limit, depth, trace, source, memory, and claim payloads are bounded.
- Branch scoring and reflection now verify branch/session ownership.
- Planner status changes and remaining budget are persisted atomically.
- Stored branch/source/critique/reflection/memory text is sanitized.
- SQLite connections now enable foreign keys, WAL, busy timeout, and restrictive new-file permissions.
- Dependency constraints now use the official `mcp` SDK, avoid the vulnerable/incompatible standalone FastMCP package surface, and remove unused `networkx`.
- Tests now cover cross-session mutation rejection, budget persistence, sanitizer hardening, research planning, and invalid MCP score rejection.

## Findings

### High: MCP tool inputs are mostly unvalidated

Affected files:

- `src/reasoning_engine/server.py`
- `src/reasoning_engine/sessions.py`
- `src/reasoning_engine/memory.py`

Examples:

- `q_score`, `advantage`, and `confidence` are documented as 0-1 values, but the
  server accepts any float.
- `depth`, `top_k`, `limit`, `trace`, `sources`, `key_learnings`, and `domain_tags`
  are not bounded.
- JSON fields are parsed but not schema-validated.
- `score_branch` accepts `session_id`, but does not verify that the branch belongs
  to that session.
- `record_reflection_tool` can reactivate any branch ID after recording a reflection.

Impact:

- A malformed or compromised MCP caller can corrupt scoring, force bad planning,
  create very large database rows, retrieve excessive memory records, or mutate a
  branch outside the stated session context.

Recommended fix:

- Add Pydantic or dataclass-based validators at the MCP boundary.
- Clamp or reject numeric scores outside `[0.0, 1.0]`.
- Enforce `0 <= depth <= session.budget_max_depth`.
- Enforce bounded list lengths and string lengths for traces, sources, critiques,
  and memory entries.
- Check `branch.session_id == session_id` before scoring or reflecting.
- Return structured error JSON for validation failures.

### High: Budget state is returned but not persisted

Affected files:

- `src/reasoning_engine/server.py`
- `src/reasoning_engine/sessions.py`
- `src/reasoning_engine/dora.py`

`select_next_branches` computes `budget_remaining`, but the sessions table is not
updated. `check_termination` then reads the original `budget_remaining_steps`, so a
session can continue indefinitely unless a high-confidence branch appears.

Impact:

- The intended resource-control mechanism does not actually stop long-running
  research loops.
- In a hostile or buggy orchestration flow, this can become a cost or availability
  problem.

Recommended fix:

- Add an atomic `apply_planning_result(session_id, result)` function that updates
  branch statuses and decrements `budget_remaining_steps` in the same transaction.
- Make `select_branches` refuse to produce negative remaining budget.
- Add regression tests for budget exhaustion after repeated planner calls.

### Medium: Dependency versions are not reproducible

Affected file:

- `pyproject.toml`

The project declares broad lower bounds:

- `fastmcp>=2.14.0`
- `networkx>=3.0`

Local MCP client tests currently fail under the installed environment because
FastMCP's pydocket/fakeredis dependency path tries to import
`fakeredis.aioredis.FakeServer`, which is unavailable in the installed fakeredis
version.

Impact:

- Fresh installs can produce different behavior from CI or the developer machine.
- MCP smoke tests can fail before application code runs.

Recommended fix:

- Add a lockfile using `uv`, Poetry, or pip-tools.
- Pin known-good production ranges, including relevant transitive constraints where
  needed.
- Add a minimal MCP client smoke test to CI that uses the locked environment.

### Medium: Prompt-injection handling is helper-level, not policy-level

Affected files:

- `src/reasoning_engine/sanitizer.py`
- `skill/deep-research.md`

The skill instructs agents to sanitize Crawl4AI content before use, but the server
does not enforce whether branch traces, source excerpts, critiques, reflections, or
memories were sanitized before persistence and later recall.

Impact:

- Unsanitized hostile source content can be stored in traces or episodic memory and
  reintroduced into future prompts through `recall_memory_tool` or
  `consensus_candidates`.

Recommended fix:

- Sanitize or classify all externally sourced text before persistence.
- Store raw and sanitized source excerpts separately only if raw retention is
  necessary; return sanitized text by default.
- Add a provenance flag such as `sanitized: true` or `source_type`.
- Add tests for indirect prompt injection via saved memory and consensus traces.

### Medium: SQLite production settings are minimal

Affected file:

- `src/reasoning_engine/db.py`

The connection enables foreign keys, but does not set WAL mode, busy timeout,
synchronous level, file permissions, schema versioning, or migrations.

Impact:

- Parallel MCP tool calls can hit lock contention.
- Schema changes will be difficult after real data exists.
- Database files can contain sensitive research queries and excerpts without
  permission hardening.

Recommended fix:

- Set `PRAGMA journal_mode=WAL`, `PRAGMA busy_timeout=5000`, and an explicit
  synchronous policy.
- Create parent directories safely for custom DB paths.
- Set restrictive file permissions on newly created database files where supported.
- Add a `schema_migrations` table or move to Alembic/sqlite-utils migrations.

### Medium: Error handling leaks internal input snippets

Affected file:

- `src/reasoning_engine/server.py`

`_parse_json` includes the first 200 characters of invalid input in the exception
message. This is helpful during development but can leak sensitive source excerpts
or prompt text into client-visible errors and logs.

Recommended fix:

- Return a field name and parse position, but omit raw input by default.
- Gate raw snippets behind a debug environment variable.

### Low: Sanitizer is regex-only and strips rather than neutralizes context

Affected file:

- `src/reasoning_engine/sanitizer.py`

The sanitizer removes common obvious patterns, but regex-only filtering will miss
many injection variants and can create misleading text by deleting words without
adding a warning.

Recommended fix:

- Add an explicit prefix to sanitized external content: "External untrusted content".
- Track detected patterns in metadata.
- Use an HTML parser such as `bleach` or `BeautifulSoup` for HTML extraction.
- Add tests for obfuscated injection markers, markdown links, encoded HTML, and
  nested/malformed tags.

### Low: Unused dependency

Affected file:

- `pyproject.toml`

`networkx` is declared but not imported in the tracked source.

Recommended fix:

- Remove it if not planned soon, or add the graph/tree functionality that requires
  it. Smaller dependency graphs reduce audit and supply-chain surface.

## Recommended Roadmap

1. Add strict MCP boundary validation and session/branch ownership checks.
2. Fix persisted budget accounting and atomic planner updates.
3. Add a lockfile and resolve the FastMCP/pydocket/fakeredis test failure.
4. Expand adversarial tests for malformed JSON, score bounds, oversized payloads,
   hostile source excerpts, and cross-session branch mutation.
5. Harden SQLite settings, permissions, and migration strategy.
6. Enforce sanitized-by-default storage and recall for externally sourced text.
7. Fix formatting/lint issues and make local test commands match CI docs.

## Suggested First Implementation Batch

The best first batch is small and high-impact:

- Add `src/reasoning_engine/validation.py` for shared constants and validation
  helpers.
- Validate all tool inputs in `server.py`.
- Add `get_branch_for_session` and use it before score/reflection/status changes.
- Persist budget decrement in one transaction with branch status updates.
- Add tests for invalid scores, invalid branch/session pairing, negative depth,
  over-limit `top_k`, malformed source JSON, and budget exhaustion.
